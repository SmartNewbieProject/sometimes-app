연동 부재: 알림 읽음 처리 미호출 OPEN
NotificationIcon은 useUnreadCount()를 통해 GET /notifications/unread-count API를 폴링합니다. 채팅 읽음 이벤트(MESSAGES_READ_REQUESTED) 발생 시 PATCH /notifications/{id}/read를 호출하거나 GET /notifications/unread-count 쿼리를 invalidate하는 로직이 존재하지 않습니다. 결과적으로 채팅 메시지를 읽어도 알림 카운트는 서버가 별도 이벤트를 보내기 전까지 갱신되지 않습니다.
해결 방향
  • 채팅방 진입 시(MESSAGES_READ_REQUESTED) 해당 방의 chat_message 타입 알림을 batch 읽음 처리
  • query-cache-manager.tsx의 markRoomAsReadInCache 내부 또는 이벤트 핸들러에서 ['notification-unread-count'] 쿼리 invalidate 추가
  • 또는 PATCH /notifications/read-by-reference 같은 batch API가 없다면, GET /notifications/unread-count refetch 트리거만으로도 단기 해결 가능

버그 2 — 내가 보낸 채팅에 삭제/수락 표시

HIGH 백엔드 프론트엔드 캐시 문제

내가 먼저 채팅을 시작했는데도 채팅방 목록에서 삭제/수락 버튼이 표시되는 문제입니다. paymentConfirm 필드가 false일 때 발신자/수신자 구분 없이 Lock 상태로 렌더링되기 때문입니다. 

// src/features/chat/ui/room-list/chat-room-card.tsx
function ChatRoomCard({ item }) {
  if (item.paymentConfirm) {
    return <OpenVariant item={item} />;  // 열린 채팅
  }
  return <LockVariant item={item} />;     // 잠긴 채팅 (삭제|수락) ← 발신자도 여기 빠짐
}
시나리오 A — 백엔드 응답 오류 CRITICAL
GET /chat/rooms/cursor 응답에서 발신자(채팅 시작자)임에도 paymentConfirm: false가 반환되는 경우. 서버의 채팅방 목록 쿼리가 발신자 여부를 고려하지 않고 단순 paymentConfirm 컬럼 값만 반환할 때 발생할 수 있습니다. 우선적으로 서버 로그 및 응답 raw 데이터를 확인해야 합니다.
시나리오 B — 캐시 미갱신 (캐시 동기화 이슈) OPEN
useChatLockQuery의 잠금해제 뮤테이션(POST /chat/rooms/{id}/enter) onSuccess에서 ['chat-room'] 쿼리를 invalidate하지 않습니다. 채팅방 입장 후 뒤로 나오면 stale 캐시에 담긴 paymentConfirm: false 상태가 그대로 표시됩니다. 재현 조건: 채팅 잠금해제 → 채팅방 입장 → 뒤로가기 → 목록 확인.
시나리오 C — 타입 구조 문제 OPEN
ChatRoomList 타입에 발신자/수신자를 구분하는 isInitiator 또는 isSender 필드가 없습니다. paymentConfirm 단일 조건으로 UI를 분기하면 발신자가 항상 수신자로 취급될 위험이 있습니다. UI 분기 로직을 paymentConfirm || isInitiator 형태로 개선하는 것이 안전합니다.
