Sometimes App 성능 의사결정 보드

현재 코드 기준으로 가장 느릴 가능성이 큰 구간은 채팅방 진입/실시간 대화 중입니다. 그다음은 홈 화면 복귀 시 fetch 묶음, 앱 첫 실행 초기화, 모든 API 요청 전 storage 조회 순입니다.

가장 느릴 가능성: 채팅 두 번째: 홈 포커스 공통 비용: axios + storage 추천 시작점: 채팅 캐시/리스트
한 줄 결론 “채팅을 먼저 줄이면 체감 속도 개선이 가장 클 가능성”

실시간 업데이트, 전체 리스트 재계산, 재조회, 전역 소켓이 한 구간에 겹쳐 있습니다.

이 문서의 용도 적용 여부를 빠르게 결정

아래 카드에서 적용 / 보류 / 미적용을 직접 눌러보면 상단 요약이 바뀝니다.

핵심 요약

측정값이 아닌 정적 분석 우선순위
1위
채팅방: 새 메시지마다 배열 재계산 + 캐시 invalidate + refetch
2위
홈: 포커스 시 heartbeat와 여러 query invalidate가 동시 실행
3위
앱 시작: 폰트, SDK, 알림, ATT, provider가 초기 진입에 집중
공통
axios 인터셉터가 요청마다 storage에서 국가/토큰을 비동기 조회

우선순위 랭킹

체감 속도 기준
1

채팅방 렌더링 + 실시간 갱신

메시지 한 건 추가될 때도 전체 메시지 배열을 다시 평탄화하고 정렬하며, 캐시 수동 갱신 뒤 다시 invalidate까지 수행합니다.

2

홈 화면 복귀 시 fetch 묶음

포커스마다 heartbeat, 알림, 매칭, 프로필, 선호도 관련 쿼리가 한 번에 갱신됩니다.

3

루트 전역 채팅/배지/활동 추적

채팅 화면이 아니어도 소켓 모듈, unread 계산, 활동 추적 query가 앱 루트에서 항상 살아 있습니다.

4

모든 API 요청 전 storage 조회

요청 인터셉터에서 국가/토큰을 AsyncStorage 기반으로 읽어 네트워크 앞단에 반복 지연이 붙습니다.

이 선택은 브라우저의 localStorage에 저장됩니다.

채팅 캐시 invalidate 축소

상태: 미선택
효과 가장 큼
영향 화면: 채팅방 난이도: 중간 리스크: 낮음~중간
왜 느린가: setQueryData 후 invalidateQueries가 겹쳐 메시지 이벤트마다 네트워크/렌더가 중복됩니다.
기대효과: 메시지 전송/수신 시 버벅임과 재조회 횟수 감소.
판단: 실시간 채팅 체감 개선 목적이면 가장 먼저 해볼 가치가 큽니다.

채팅 리스트 전체 재계산 줄이기

상태: 미선택
효과 큼
영향 화면: 채팅방 난이도: 중간~높음 리스크: 중간
왜 느린가: 메시지 배열 평탄화, 정렬, 날짜 divider 생성이 새 메시지 때마다 전체 재실행됩니다.
기대효과: 대화가 길수록 개선 폭이 커집니다.
판단: 채팅방이 핵심 KPI라면 invalidate 축소와 세트로 진행하는 편이 좋습니다.

홈 포커스 refetch 정리

상태: 미선택
효과 중상
영향 화면: 홈 난이도: 낮음~중간 리스크: 낮음
왜 느린가: 홈 복귀 때 heartbeat와 여러 invalidate가 동시에 붙습니다.
기대효과: 홈 복귀 체감 속도 개선, 네트워크 burst 감소.
판단: 구현 대비 효과가 좋아서 채팅 다음 우선순위로 적절합니다.

axios 인터셉터 storage 캐싱

상태: 미선택
효과 중간
영향 범위: 전역 난이도: 중간 리스크: 중간
왜 느린가: 요청마다 국가/토큰 조회가 비동기로 반복됩니다.
기대효과: 전체 API 체인의 앞단 지연 감소.
판단: 체감은 분산되지만 구조적으로 꼭 정리할 가치가 있습니다.

전역 채팅 모듈 지연 마운트

상태: 미선택
효과 중간
영향 범위: 루트 난이도: 높음 리스크: 중간~높음
왜 느린가: 채팅 화면이 아니어도 소켓, unread 계산, activity tracking이 상시 살아 있습니다.
기대효과: 앱 전체 배경 비용 감소.
판단: 구조 리팩터링 성격이라 빠른 체감 개선 다음에 진행하는 편이 안전합니다.

앱 시작 초기화 분산

상태: 미선택
효과 선택적
영향 화면: 앱 시작 난이도: 중간 리스크: 중간
왜 느린가: 폰트, SDK, ATT, 알림 처리, provider가 첫 렌더 근처에 몰려 있습니다.
기대효과: cold start 개선 가능.
판단: 현재 가장 큰 체감 병목은 채팅이므로 1차 대상은 아닙니다.

지연이 쌓이는 흐름

사용자 여정 기준
앱 시작
폰트 로딩, SDK 초기화, 알림/ATT 처리, 여러 전역 provider가 동시에 붙습니다. 첫 인상 속도에 영향이 있지만, 지금 코드 기준 최우선 병목은 아닙니다.
홈 진입/복귀
heartbeat + 여러 invalidate가 같이 실행되고, 배너/리뷰 슬라이더와 플로팅 카드가 함께 렌더링됩니다. 복귀 직후의 네트워크 burst가 체감 버벅임으로 이어질 수 있습니다.
채팅방 진입
채팅 쿼리가 항상 refetch되고, 메시지 목록을 다시 계산합니다. 방 진입과 동시에 실시간 연결, room join, read 처리까지 겹칩니다.
대화 중
새 메시지마다 캐시 수정과 재조회, 전체 정렬/구분선 재생성, 이미지 업로드 시 추가 작업이 붙습니다. 가장 큰 체감 병목 후보입니다.

권장 실행 순서

빠른 체감 개선 우선
1단계

채팅 캐시 invalidate 축소

메시지 수신/전송 성공 후의 중복 refetch를 제거하고, UI가 캐시 갱신만으로 유지되도록 바꿉니다.

추천 이유: 효과 대비 위험이 가장 좋음
2단계

채팅 리스트 재계산 최소화

정렬과 날짜 divider 생성을 매 렌더 전체 수행하지 않도록 구조를 바꿉니다.

추천 이유: 대화가 길수록 개선 폭이 큼
3단계

홈 포커스 갱신 묶음 정리

invalidate 수를 줄이거나 staleTime 전략을 조정해 홈 복귀 시 burst를 완화합니다.

추천 이유: 구현 난이도 낮고 체감이 분명함
4단계

axios/storage 캐싱, 루트 전역 비용 정리

전역 구조 비용을 줄여 앱 전체의 기본 반응성을 끌어올립니다.

추천 이유: 구조 개선 성격, 후순위가 안전
