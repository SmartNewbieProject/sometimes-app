# 프로필 조회자 확인 기능 사업성 분석 보고서

## Executive Summary

프로필 조회자 확인 기능은 **중상위 사업 가치(총점 74/100)**를 가진 검증된 수익화 모델이다. 경쟁사 4곳 전부 유사 기능을 운영 중이며, 특히 틴더의 'Likes You'는 프리미엄 구독 전환의 핵심 동인으로 작용한다. 썸타임의 차별화 포인트인 **'여러 번 조회' 시그널 강조**와 **학교 정보 힌트 제공**은 대학생 타겟에 최적화된 전략이다. 단, 구현 난이도가 중간(서버 로그 인프라 필요)이고 여성 유저 반발 리스크가 존재하여 **Phase 2 우선순위 권장**(핵심 매칭 품질 개선 후 도입).

## 1. 핵심 발견 (Key Findings)

### 1.1 시장 수요 검증 (근거: 경쟁사 벤치마크)

**결론: 데이팅 앱 필수 기능으로 자리잡음 (4/4 경쟁사 보유)**

| 경쟁사 | 기능명 | 과금 모델 | 시장 반응 |
|--------|---------|-----------|----------|
| **틴더** | Likes You | Gold 구독 (월 $14.99) | 구독 전환 핵심 동인 (출처: PRD 벤치마크) |
| **아만다** | 내게 관심 있는 사람 | 아이템 과금 (5리본) | 블러 처리 후 공개 방식 검증됨 |
| **글램** | 호감 확인 | 다이아 티어 무제한 | VIP 혜택으로 활용 |
| **위피** | 내 프로필 조회자 | 유료 구독 | 조회 기록만 제공 (썸타임보다 얕은 기능) |

**출처**: `/docs/research/product/prd-profile-viewer.md` 섹션 10

**경쟁사 분석 인사이트**:
- 틴더는 이 기능을 **구독 모델의 killer feature**로 활용
- 국내 앱들은 **단건 결제(Pay-to-Reveal)** 방식 선호 (썸타임의 5구슬 전략과 일치)
- 위피조차 제공하는 기능을 썸타임이 없으면 **기능 경쟁력 열위** 인식

### 1.2 수익 기여도 추정 (근거: 내부 목표 + 경쟁사 유추)

**PRD 목표**: 월 10만 구슬 소비 (= 구슬당 188원 기준 **월 1,880만 원**)

**검증 로직**:
```
현재 DAU 6,000명 기준
- 기능 사용률 30% 달성 시 → 1,800명/일
- 이 중 20%가 구슬 사용 → 360건/일
- 360건 × 5구슬 × 30일 = 54,000 구슬/월
- 54,000 × 188원 = **월 1,015만 원**
```

**현실성 평가**:
- 목표(10만 구슬)는 **사용률 56%** 필요 → 다소 공격적
- 보수 추정치(5만 구슬/월 940만 원)가 현실적
- 그러나 현재 LTV 6,150원 대비 **153% 증가** 효과로 BM 개선에 크게 기여

**출처**: `/docs/research/competitors/sometime-vs-top-competitors.md` LTV:CAC 섹션

### 1.3 전략 정합성 (근거: 썸타임 핵심 가치)

**높은 정합성 (5/5점)**

| 썸타임 브랜드 가치 | 기능 연결 |
|-------------------|----------|
| **설렘** | "누군가 나를 여러 번 봤다" → 두근거림 유발 |
| **진정성** | 상호 관심 기반 연결 (일방적 매칭 X) |
| **성장** | 먼저 다가가는 용기를 낼 수 있는 계기 |

**심리학적 근거**:
> "누가 나를 조회했나요?" 기능은 **호기심 갭(Curiosity Gap)**을 활용한 행동심리학 전략이다. 3일 대기 기간 중에도 "OO대학교 학생이 회원님의 프로필을 흥미롭게 봤어요"라는 블러 처리된 알림으로 푸시 알림 반응률을 극대화할 수 있다.

**출처**: `/docs/research/user-research/z-gen-habit-forming-platform.md` 섹션 3

### 1.4 경쟁 차별화 포인트

**썸타임만의 강점 3가지**:

1. **"여러 번 확인" 시그널 강조** (단순 1회 조회 X)
   - 경쟁사: 조회 여부만 표시
   - 썸타임: 2회 이상 + 24시간 내 패턴 감지 → **진성 관심도 필터링**

2. **학교 정보 힌트 제공**
   - 블러 처리 시에도 학교명/나이 표시
   - 대학생 타겟에 최적화: "우리 학교 사람인가?" 궁금증 유발

3. **환불 보장 정책**
   - 탈퇴/휴면 유저 확인 시 100% 환불
   - 경쟁사 대비 **신뢰 구축** (초기 스타트업에 중요)

**출처**: `/docs/research/product/prd-profile-viewer.md` 섹션 10

## 2. 사업성 매트릭스 평가

### 평가 체계
- **시장 수요** (25점): 경쟁사 보유율, 유저 니즈 검증도
- **경쟁 차별화** (20점): 썸타임만의 독특한 가치
- **수익 기여도** (20점): 매출 증대 잠재력
- **구현 난이도** (15점): 기술적 복잡도 (낮을수록 높은 점수)
- **전략 정합성** (20점): 브랜드 가치와의 일치도

### 점수표

| 평가 항목 | 배점 | 득점 | 근거 |
|----------|------|------|------|
| **시장 수요** | 25 | **22** | 4/4 경쟁사 보유, 틴더 구독 핵심 기능 (-3: 국내 시장 검증 부족) |
| **경쟁 차별화** | 20 | **17** | 여러 번 조회 강조, 학교 힌트, 환불 정책 차별화 (-3: 위피도 유사 기능 보유) |
| **수익 기여도** | 20 | **14** | LTV 153% 증가 잠재력 (-6: 목표 10만 구슬은 과대, 5만이 현실적) |
| **구현 난이도** | 15 | **9** | DB 스키마 3개 추가, 푸시 알림 로직 필요 (-6: 조회 로그 인프라 부재 시 고비용) |
| **전략 정합성** | 20 | **12** | 설렘/진정성/성장 가치 일치 (-8: 여성 유저 스토킹 우려 리스크) |
| **총점** | **100** | **74** | **중상위 사업 가치** |

### 점수 해석

- **80점 이상**: 즉시 개발 착수 권장
- **60-79점**: Phase 2 우선순위 (선행 조건 충족 후)
- **40-59점**: 보류 또는 Pivot 필요
- **40점 미만**: 기각 권장

**결론**: 74점은 **Phase 2 우선순위**에 해당. 핵심 매칭 품질 개선 후 도입 권장.

## 3. 리스크 및 대응 전략

### 3.1 여성 유저 반발 리스크 (영향도: 높음)

**문제**:
- 프로필 조회자 기능은 본질적으로 **정보 비대칭** 해소
- 여성 유저는 "누가 나를 봤는지 알려주는 게 스토킹 느낌" 우려 가능
- 현재 썸타임 핵심 과제는 **여성 유저 확보**

**대응 방안** (PRD 제시):
1. 조회 횟수 최대 "5회 이상"까지만 표시 (정확한 수치 숨김)
2. 차단 연동: 차단한 사용자 조회 기록 비표시
3. **여성 대상 첫 확인 무료** 혜택 제공

**추가 제안**:
- 베타 테스트를 **대전 10% 여성 유저** 대상 A/B 테스트로 진행
- 거부감 측정: NPS 설문 "이 기능이 불편하셨나요?" 수집

**출처**: `/docs/research/product/prd-profile-viewer.md` 섹션 9

### 3.2 유령 프로필 확인 리스크 (영향도: 중간)

**문제**:
- 구슬 5개 사용 → 확인한 유저가 이미 탈퇴/휴면 상태
- **"돈만 날렸다"** 불만 → CS 이슈 + 앱 신뢰도 하락

**대응 방안**:
- PRD 명시된 **100% 환불 정책** 철저히 이행
- 조회자 목록에서 **최근 7일 내 미접속 유저 자동 제외**
- UI에 "마지막 접속: 2시간 전" 정보 추가 (투명성 확보)

### 3.3 구현 복잡도 (영향도: 중간)

**기술 요구사항**:
```sql
-- 3개 테이블 신규 생성 필요
1. profile_views (조회 로그)
2. profile_viewer_summary (집계 테이블)
3. gem_usage_log (구슬 사용 로그)
```

**병목 지점**:
- 현재 백엔드에 **조회 로그 수집 인프라 부재** 시 구축 비용 발생
- 푸시 알림 트리거 배치 잡 개발 필요

**대응 방안**:
- Phase 1 내부 테스트 시 **부하 테스트** 필수
- AWS CloudWatch로 조회 로그 급증 모니터링

**출처**: `/docs/research/product/prd-profile-viewer.md` 섹션 6

### 4.2 Phase 2 개발 (2-3개월 후)

**전제 조건**:
1. ✅ 핵심 매칭 품질 개선 완료 (추천 알고리즘 고도화)
2. ✅ 여성 유저 비율 30% 이상 확보
3. ✅ DAU 10,000명 이상 달성

**개발 순서**:
```
Week 1-2: DB 스키마 설계 + 조회 로그 수집 인프라 구축
Week 3-4: API 개발 (조회자 목록 조회, 구슬 사용)
Week 5-6: 푸시 알림 트리거 개발 + UI 구현
Week 7: 내부 테스트 (팀 전체)
Week 8: 대전 10% 소프트 런칭 (A/B 테스트)
Week 9-10: 데이터 분석 + 개선
Week 11: 전체 오픈
```

### 4.3 성공 지표 모니터링 (KPI Dashboard)

**1차 지표 (기능 사용)**:
- 기능 사용률: DAU 중 조회자 목록 진입 **≥ 30%**
- 구슬 사용률: 목록 진입 → 프로필 공개 **≥ 20%**
- 좋아요 전환율: 프로필 공개 → 좋아요 발송 **≥ 25%**

**2차 지표 (비즈니스 임팩트)**:
- 월 구슬 소비량: **50,000 구슬** (보수 목표)
- DAU 재방문율: **+5%p** (리텐션 기여)
- 매칭 기여율: 전체 매칭 중 **10%**가 조회자 기능 경유

**3차 지표 (리스크 모니터링)**:
- 여성 유저 NPS: **≥ 60점** (스토킹 우려 최소화)
- CS 문의 비율: 전체 문의 중 **< 5%** (환불 이슈 통제)

## 5. 대안 시나리오 (Option B)

### 만약 여성 유저 반발이 강하다면?

**Pivot 1: "양방향 조회자 확인"**
- 내가 확인한 프로필도 상대에게 알림 (상호 투명성)
- 심리적 부담 감소: "어차피 서로 보는 거잖아"

**Pivot 2: "무료 버전 제공"**
- 조회자 수만 무료 공개 ("3명이 확인했어요")
- 구체적 프로필은 구슬 사용 (단계적 과금)

**Pivot 3: "매칭 확률 정보로 전환"**
- 조회자 신원 대신 "매칭 가능성 90% 이성 3명 대기 중" 표시
- 프라이버시 우려 완화 + 호기심 유지
